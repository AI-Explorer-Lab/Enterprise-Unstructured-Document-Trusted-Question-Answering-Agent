# 重点聚焦

本文档面向面试复盘，重点围绕项目中最能体现 RAG 工程能力、Agent 编排能力和可信问答能力的模块展开。

项目定位可以先用一句话讲清楚：

> 这是一个企业非结构化 PDF 的可信问答 Agent。它把 PDF 解析成带标题、页码、表格上下文的结构化 chunk，再通过 dense + BM25 + table-aware 的混合检索召回证据，经过 rerank、去重和 evidence gate 后，才生成带 citation、trace 和 confidence 的答案。

重点模块包括：

- 混合检索模块
- Rerank 与去重模块
- Trusted QA Agent 编排模块
- 技能路由模块
- 证据门控与可信回答模块
- 数据模型、会话与 Trace 模块

## 1. 混合检索模块

### 模块介绍

混合检索模块负责从已经入库的 PDF chunks 中召回候选证据。它不是单一路径的向量检索，而是把多种检索方式组合起来：

- dense retrieval：基于 embedding 的语义相似度检索。
- BM25 sparse retrieval：基于关键词匹配的稀疏检索。
- table-aware retrieval：当问题与表格、指标、数值、财务数据有关时，单独检索表格 chunk。
- query expansion：对原始问题扩展多个检索 query，提高召回覆盖面。
- retrieval cache：对相同问题、集合、类型、top_k 的检索结果做 TTL 缓存。

核心代码位置：

- `service/retrieval/hybrid_retriever.py`
  - `HybridRetriever.retrieve(...)`
- `service/retrieval/parallel_query_executor.py`
  - `ParallelQueryExecutor.execute(...)`
  - `ParallelQueryExecutor._build_query_variants(...)`
  - `ParallelQueryExecutor._run_route_task(...)`
  - `ParallelQueryExecutor._execute_route_sync(...)`
  - `ParallelQueryExecutor._merge_candidates(...)`
- `service/retrieval/pgvector_repository.py`
  - `PgvectorRepository.dense_search(...)`
  - `PgvectorRepository.keyword_search(...)`
  - `PgvectorRepository.table_search(...)`
  - `deterministic_embedding(...)`
- `service/retrieval/sparse_retriever.py`
  - `SparseBM25Retriever`
  - `coarse_tokenize(...)`
- `service/retrieval/retrieval_cache.py`
  - `RetrievalResultCache`

### 为什么用这个模块

企业 PDF 问答里，单纯向量检索容易有几个问题：

- 对专有名词、指标名、表格字段、页码这类精确条件不够敏感。
- 对财报、合同、制度文档中的表格数据召回不稳定。
- 用户问题通常表达不完整，直接用原问题检索容易漏掉相关 chunk。
- 一次问答可能重复检索相同问题，需要缓存减少延迟。

所以项目采用混合检索：dense 负责语义召回，BM25 负责关键词精确召回，table route 负责表格证据优先召回，query expansion 负责拓宽召回面。

面试可以这样说：

> 我没有只用向量检索，因为企业文档里大量问题是指标、页码、条款、表头这种精确匹配场景。dense 能解决语义相似，BM25 能补关键词命中，table route 能保证表格问答不会被普通文本 chunk 淹没。

### 怎么用的

在 QA 主流程里，`TrustedQAWorkflow.ask(...)` 会调用：

```python
retrieval_result = await self.retriever.retrieve(
    question=question,
    collection_name=collection_name,
    top_k=top_k,
    query_type=query_type,
    expand_query_num=expand_query_num,
    enable_cache=enable_cache,
    expanded_queries=expanded,
)
```

`HybridRetriever.retrieve(...)` 内部先让 `ParallelQueryExecutor.execute(...)` 并发跑多路召回，再把候选结果交给 `TwoStageHybridReranker.rerank(...)`。

简化链路：

```text
用户问题
  -> query expansion
  -> dense route
  -> BM25 route
  -> table route（表格类问题启用）
  -> merge candidates
  -> rerank
  -> selected evidence
```

`ParallelQueryExecutor` 里会先构造 query variants：

```text
原问题 + 扩展问题列表
```

然后对每个 variant 并发执行：

```text
dense search
BM25 search
table search（按条件启用）
```

最后 `_merge_candidates(...)` 会把多路结果按 `chunk_id` 合并，把来源记录到 `source_channels`，例如同一个 chunk 可能同时来自 dense 和 bm25。

### 有什么效果

- 提高召回率：同一个问题可以从语义、关键词、表格三个角度召回。
- 提高稳定性：LLM 或 embedding 表达不准时，BM25 仍能命中关键字。
- 支持表格问答：对 `table_qa` 或看起来像表格问题的 query，会额外跑 table route。
- 便于调试：`retrieval_trace` 记录 query variants、task trace、cache hit、候选数量等信息。

### 面试可讲亮点

可以这样回答：

> 混合检索模块的核心是并发多路召回。系统先对用户问题做 query expansion，然后每个 query 同时走 dense、BM25 和必要时的 table route。召回结果不是直接返回，而是 merge 成候选池，保留每个 chunk 的来源通道，后续 rerank 会综合 dense score、BM25 score、metadata boost 和 table boost 做排序。

## 2. Rerank 与去重模块

### 模块介绍

Rerank 与去重模块负责把混合检索返回的大量候选 chunk 重新排序，并去掉重复或近重复内容，最终选出 top-k evidence。

核心代码位置：

- `service/retrieval/two_stage_hybrid_reranker.py`
  - `TwoStageHybridReranker.rerank(...)`
  - `_metadata_boost(...)`
  - `_table_boost(...)`
  - `_normalize_exact_text(...)`
  - `_near_duplicate_score(...)`
  - `TwoStageHybridReranker._finalize_selection(...)`
  - `TwoStageHybridReranker._supplement_neighbors(...)`

### 为什么用这个模块

混合检索召回的是候选池，不是最终答案依据。候选池里常见问题包括：

- dense 和 BM25 可能召回同一个 chunk。
- 不同 chunk 内容非常相似，重复占用 top-k 名额。
- 向量分高的 chunk 不一定来自正确章节。
- 表格类问题如果不做表格加权，表格 chunk 可能排不上来。
- 单个 chunk 可能缺上下文，需要补充相邻 chunk。

所以 rerank 模块要做二次排序和证据清洗。

### 怎么用的

`HybridRetriever.retrieve(...)` 在完成并发检索后调用：

```python
reranked, rerank_trace = self.reranker.rerank(
    query=question,
    candidates=candidates,
    top_k=top_k,
    query_type=query_type,
    table_evidence_quota=self.table_evidence_quota,
)
```

核心打分逻辑在 `TwoStageHybridReranker.rerank(...)` 中，最终分数由四部分构成：

```text
final_score =
  dense_weight * dense_score
  + bm25_weight * bm25_score
  + metadata_boost_weight * metadata_boost
  + table_boost_weight * table_boost
```

各部分含义：

- `dense_score`：语义相似度。
- `bm25_score`：关键词匹配度。
- `metadata_boost`：标题路径、一级标题、二级标题、三级标题、文档名、页码 hint 的加权。
- `table_boost`：表格类问题中，对 table chunk、table header、table context 的加权。

去重逻辑主要有两层：

- 精确文本去重：通过 `_normalize_exact_text(...)` 把文本归一化后判断是否重复。
- 近重复去重：通过 `_near_duplicate_score(...)` 比较 token set 的 containment / Jaccard 相似度。

后处理逻辑：

- `_supplement_neighbors(...)`：补充相邻 chunk，避免答案缺上下文。
- `_finalize_selection(...)`：保证表格证据配额，控制最终 top-k。

### 有什么效果

- 候选证据更干净，减少重复 chunk。
- 排序更符合问题意图，而不是只看向量分。
- 表格问题能优先拿到表格证据。
- 保留 `rerank_trace`，可以解释为什么某些 chunk 被选中。

### 面试可讲亮点

可以这样回答：

> Rerank 阶段不是简单重排，而是融合 dense、BM25、metadata、table prior 的加权排序。比如用户问“某指标 2023 年是多少”，metadata boost 会考虑标题路径和页码 hint，table boost 会优先表格 chunk。之后再做精确去重和近重复去重，保证 top-k 证据多样且相关。

## 3. Trusted QA Agent 编排模块

### 模块介绍

Trusted QA Agent 编排模块是整个问答链路的主控制器。它负责把 query classification、skill selection、clarify gate、retrieval、rerank、evidence gate、answer generation、session save 串起来。

核心代码位置：

- `service/agent/trusted_qa_workflow.py`
  - `TrustedQAWorkflow`
  - `TrustedQAWorkflow.ask(...)`
  - `TrustedQAWorkflow._build_response(...)`
  - `TrustedQAWorkflow._save(...)`
- `service/agent/query_classifier.py`
  - `classify_query_type(...)`
- `service/agent/query_expander.py`
  - `expand_queries(...)`
- `service/agent/clarify_gate.py`
  - `run_clarify_gate(...)`
  - `extract_slots(...)`

### 为什么用这个模块

RAG 问答系统如果只写成“检索 + 生成”，后续很难扩展：

- 不同问题类型需要不同处理方式。
- 有些问题缺少指标、年份、对比对象，需要先追问。
- 证据不足时不能直接让 LLM 编答案。
- 需要把检索、重排、门控、生成过程记录到 trace，方便排查和解释。

因此项目用 `TrustedQAWorkflow` 做集中编排，把每个阶段显式化。

### 怎么用的

API 入口在 `controller/apis/qa_controller.py`：

```python
@router.post("/qa/ask")
async def ask(request: QARequest):
    return await get_trusted_qa_workflow().ask(...)
```

`TrustedQAWorkflow.ask(...)` 的主流程可以这样理解：

```text
load_session
  -> classify_query_type
  -> select_skill
  -> query expansion
  -> clarify gate
  -> hybrid retrieval
  -> evidence gate
  -> retry retrieval if needed
  -> answer generation
  -> optional LLM grounded answer
  -> evaluation
  -> save session and trace
```

关键阶段：

- `classify_query_type(question)`：判断问题类型，例如 fact lookup、table QA、summarization、citation locate、report generation、multi-doc compare。
- `self.skill_registry.select_skill(query_type)`：根据问题类型选择技能。
- `run_clarify_gate(...)`：检查问题是否缺少必要槽位。
- `self.llm_service.expand_queries(...)`：优先用 LLM 扩展 query。
- `expand_queries(...)`：LLM 不可用时用模板扩展。
- `self.retriever.retrieve(...)`：执行混合检索和 rerank。
- `self.evidence_gate.evaluate(...)`：证据门控。
- `self.answer_generator.generate(...)`：生成本地模板答案和 citation。
- `self.llm_service.generate_grounded_answer(...)`：在证据通过时用 LLM 生成最终自然语言答案。
- `self._build_response(...)`：组装 answer、decision、citations、evidence、retrieval_trace、rerank_trace、skill_trace。
- `self._save(...)`：保存会话和 trace。

### 有什么效果

- 问答流程可解释：每一步都能记录到 trace。
- 问答更可信：先过 evidence gate，再生成答案。
- 支持追问：缺槽位时返回 clarify，而不是盲答。
- 支持重试：证据不足时自动 retry retrieval。
- 支持技能扩展：新增 query type 和 skill 时，不需要重写整个 QA 流程。

### 面试可讲亮点

可以这样回答：

> TrustedQAWorkflow 是这个项目的核心编排层。它把 RAG 流程拆成可观测的阶段：分类、技能选择、澄清、检索、重排、证据门控、生成和保存。这样做的好处是每一步都有 trace，证据不足不会直接生成答案，而是重试、追问或拒答。

## 4. 技能路由模块

### 模块介绍

技能路由模块负责把不同类型的问题路由到不同的技能定义。这里的 skill 不是独立 Agent，而是对不同 query type 的处理策略、必需槽位、工具链、输出 schema 和 guardrails 的声明。

核心代码位置：

- `service/agent/skills.py`
  - `SkillDefinition`
  - `FactLookupSkill`
  - `TableQASkill`
  - `CitationLocateSkill`
  - `SummarizationSkill`
  - `ReportGenerationSkill`
  - `MultiDocCompareSkill`
- `service/agent/skill_registry.py`
  - `SkillRegistry`
  - `SkillRegistry.select_skill(...)`
- `service/agent/query_classifier.py`
  - `classify_query_type(...)`
- `service/agent/clarify_gate.py`
  - `required_slots_for_query_type(...)`
  - `find_missing_slots(...)`

### 为什么用这个模块

企业文档问答不是只有一种问题：

- 查事实：直接找相关条款或片段。
- 查表格：需要指标、年份、单位、表头。
- 找出处：重点是页码、chunk_id、引用片段。
- 做总结：需要范围和多条证据覆盖。
- 生成报告：需要结构化输出。
- 多文档对比：需要至少两个文档或对象。

如果所有问题都走同一个 prompt 和检索策略，容易出现回答不稳定、证据要求不一致的问题。

技能路由的价值是：把不同问题类型对应的处理约束显式写出来。

### 怎么用的

`query_classifier.py` 先基于关键词和规则识别问题类型：

```text
表格 / 指标 / 同比 / 环比 -> table_qa
总结 / 概述 / 摘要 -> summarization
出处 / 原文 / 哪一页 / 引用 -> citation_locate
报告 / 分析报告 -> report_generation
对比 / 比较 / 差异 -> multi_doc_compare
默认 -> fact_lookup
```

然后 `SkillRegistry.select_skill(query_type)` 根据 query type 返回技能定义。

每个 `SkillDefinition` 包含：

- `skill_name`：技能名。
- `query_types`：适配的问题类型。
- `required_slots`：必填槽位。
- `input_schema`：输入结构。
- `tool_chain`：工具链。
- `output_schema`：输出结构。
- `guardrails`：该技能的约束。
- `trace_fields`：需要记录的 trace 字段。

例如：

- `TableQASkill` 要求 `metric` 和 `period`。
- `MultiDocCompareSkill` 要求 `compare_targets`。
- `CitationLocateSkill` 要求 `target_statement`。
- `SummarizationSkill` 和 `ReportGenerationSkill` 要求 `scope`。

### 有什么效果

- 问题类型和处理策略解耦。
- 新增技能时只需要新增 `SkillDefinition` 并注册到 `ALL_SKILLS`。
- 不同技能可以有不同 evidence gate 要求。
- trace 中能看到 selected skill 和 tool chain，方便解释 Agent 为什么这样处理。

### 面试可讲亮点

可以这样回答：

> 我把技能抽象成配置化的 SkillDefinition，不同 query type 对应不同 required slots、tool chain 和 guardrails。比如表格问答要求 metric 和 period，多文档对比要求 compare targets。这样比把所有逻辑写进一个大 prompt 更可控，也更容易扩展。

## 5. 证据门控与可信回答模块

### 模块介绍

证据门控与可信回答模块负责判断“当前证据是否足以回答”，并基于证据生成答案、引用和置信度。

核心代码位置：

- `service/agent/evidence_gate.py`
  - `EvidenceGate`
  - `EvidenceGate.evaluate(...)`
  - `run_evidence_gate(...)`
- `service/agent/answer_generator.py`
  - `build_evidence_payload(...)`
  - `build_citations(...)`
  - `AnswerGenerator`
  - `AnswerGenerator.generate(...)`
- `service/llm/llm_client.py`
  - `LLMService.generate_grounded_answer(...)`
- `service/evaluation/ragas_evaluator.py`
  - `evaluate_qa_result(...)`

### 为什么用这个模块

可信 QA 的关键不是“能不能生成答案”，而是“有没有足够证据支撑答案”。如果没有证据门控，RAG 系统很容易：

- 检索结果很弱但仍然强行回答。
- 表格问题没有表格证据却从普通文本猜数值。
- 多文档对比只命中一个文档却输出对比结论。
- citation 不可靠，无法追溯来源。

因此项目在答案生成前加入 Evidence Gate。

### 怎么用的

`EvidenceGate.evaluate(...)` 会根据 query type 做不同判断：

通用判断：

- 没有 evidence：retry 或 refuse。
- top score 低于阈值：retry 或 refuse。
- avg score 低于阈值：retry 或 refuse。

表格问答：

- `query_type == "table_qa"` 时检查 table evidence 数量。
- 如果表格证据不足，会优先 retry。
- 如果还缺 metric 或 period，会 clarify。

多文档对比：

- `query_type == "multi_doc_compare"` 时检查是否覆盖至少两个文档。
- 覆盖不足会 retry 或 clarify/refuse。

总结和报告：

- `summarization`、`report_generation`、`multi_doc_compare` 属于 coverage-sensitive 类型，需要足够证据覆盖。

门控结果可能是：

```text
answer   -> 证据足够，可以回答
retry    -> 证据不足，重新检索
clarify  -> 问题缺关键信息，需要追问
refuse   -> 证据仍不足，拒绝回答
```

答案生成分两层：

1. `AnswerGenerator.generate(...)` 先基于 evidence 构造本地答案、evidence payload 和 citations。
2. 如果 decision 是 `answer`，再调用 `LLMService.generate_grounded_answer(...)`，要求 LLM 只能基于 evidence 回答，并且关键结论必须带 citation id。

### 有什么效果

- 降低幻觉：证据不足不会直接回答。
- 可追溯：每条 evidence 会生成 citation。
- 可解释：`gate_decision` 会写入 retrieval trace 和 skill trace。
- 支持自动补救：低证据时先 retry，不是一上来就拒答。
- 对表格、多文档场景有专门约束。

### 面试可讲亮点

可以这样回答：

> Evidence Gate 是项目里保证可信度的关键模块。它会根据问题类型检查证据质量，比如 top score、avg score、文档覆盖数、表格证据数。如果证据不足，系统会先 retry 检索，仍不足就 clarify 或 refuse。只有证据通过时，才会进入 grounded answer generation，并且每个关键结论都要带 citation。

### 为什么用这个模块

RAG 系统如果没有统一数据模型，会导致：

- API 响应字段不稳定。
- 检索候选、证据、引用之间难以关联。
- 无法复盘一次回答为什么这么答。
- 难以保存会话历史。
- 难以做评估和问题排查。

这个项目通过 Pydantic model 和 session service，把 QA 的输入输出、证据和 trace 结构化。

### 怎么用的

`QARequest` 定义 `/qa/ask` 的入参：

- `question`
- `session_id`
- `collection_name`
- `top_k`
- `expand_query_num`
- `enable_cache`

`QAResponse` 定义可信问答响应：

- `answer`
- `decision`
- `query_type`
- `confidence`
- `citations`
- `evidence`
- `retrieval_trace`
- `rerank_trace`
- `session_id`

`Chunk` 定义入库后的文档片段：

- 文档维度：`doc_id`、`collection_name`、`doc_source`
- 页码维度：`page_idx`、`page_range`
- 结构维度：`heading_path`、`level1_title`、`level2_title`、`level3_title`
- 表格维度：`table_id`、`table_header_text`、`table_context_text`
- 检索维度：`content`、`search_text`、`embedding`

`RetrievalTrace` 记录检索过程：

- 原问题
- query type
- expanded queries
- dense candidates
- sparse candidates
- merged candidates
- selected candidates
- rerank trace
- latency

`Citation` 和 `Evidence` 负责把最终答案和原始 chunk 关联起来：

- citation 里有 `chunk_id`、`doc_id`、`doc_source`、`page_idx`、`heading_path`、`quote`。
- evidence 里有 `content`、`score`、`rank` 和 citation。

`SessionService` 当前是轻量级内存实现：

- `load_session(...)`：加载或创建 session。
- `save_session(...)`：保存用户消息、助手回答和 retrieval trace。
- `get_session(...)`：根据 session_id 查询历史。
- `upsert_collection_chunks(...)`：保存 collection 下的 chunks。

数据库 schema 中已经设计了 pgvector 持久化表，包括文档、chunk、会话、消息、检索 trace 和评估记录。

### 有什么效果

- API 输出结构稳定，前端或调用方容易消费。
- 每次问答可以通过 session_id 追踪。
- 每次回答可以追溯到 evidence 和 citation。
- retrieval trace 和 rerank trace 便于 debug。
- 后续可以从内存 session 平滑迁移到数据库持久化。

### 面试可讲亮点

可以这样回答：

> 我没有只返回一段 answer，而是把 answer、decision、confidence、citations、evidence、retrieval_trace、rerank_trace、skill_trace 都结构化返回。这样系统不只是能回答，还能解释回答依据、检索路径和门控决策，便于企业场景做审计和问题排查。

## 7. 重点链路串联

可以把上面几个模块串成一条主线讲：

```text
/qa/ask
  -> QARequest
  -> TrustedQAWorkflow.ask
  -> load session
  -> classify query type
  -> select skill
  -> clarify gate
  -> query expansion
  -> HybridRetriever.retrieve
  -> ParallelQueryExecutor.execute
  -> dense + BM25 + table route
  -> merge candidates
  -> TwoStageHybridReranker.rerank
  -> EvidenceGate.evaluate
  -> retry / clarify / refuse / answer
  -> AnswerGenerator.generate
  -> optional LLM grounded answer
  -> evaluate_qa_result
  -> save session and retrieval trace
  -> QAResponse
```

面试时可以压缩成 30 秒版本：

> 用户提问后，系统先识别问题类型并选择技能。如果问题缺少指标、年份、对比对象等关键信息，会先澄清。否则系统会做 query expansion，然后走 dense、BM25 和 table-aware 的混合检索，召回候选 chunk。候选会经过 rerank 和去重，综合向量分、关键词分、metadata boost 和 table boost 得到 evidence。Evidence Gate 会判断证据是否足够，不够就重试、追问或拒答。证据通过后，系统才生成带 citation 的 grounded answer，并把 retrieval trace、rerank trace、skill trace 保存到 session。

## 9. 高频面试问答模板

### Q1：为什么不用单纯向量检索？

答：

> 因为企业 PDF 的问题经常涉及指标名、表格字段、页码、条款原文等精确匹配场景。单纯向量检索对这些信息不够稳定，所以我用了 dense + BM25 + table-aware retrieval。dense 负责语义，BM25 补关键词，table route 保证表格证据召回。

### Q2：Rerank 怎么做的？

答：

> Rerank 会把 dense score、BM25 score、metadata boost、table boost 做加权融合。metadata boost 会看标题路径、文档名和页码 hint，table boost 会在 table QA 中提高表格 chunk 权重。之后再做精确去重和近重复去重，最终输出高质量 top-k evidence。

### Q3：怎么降低幻觉？

答：

> 我把 evidence gate 放在答案生成前。它会检查 evidence 数量、top score、avg score、文档覆盖数，以及表格问题的 table evidence 数量。如果证据不足，会 retry、clarify 或 refuse。只有证据通过时才会进入 grounded answer generation。

### Q4：技能路由有什么意义？

答：

> 不同问题类型需要不同约束。比如表格问答需要 metric 和 period，多文档对比需要 compare targets，引用定位需要 target statement。用 SkillDefinition 把 required slots、tool chain、guardrails 显式配置出来，比一个大 prompt 更可控，也方便扩展。

### Q5：Trace 记录了什么？

答：

> Trace 记录了 query expansion、检索 task、候选池大小、rerank 权重和 top evidence、evidence gate 决策、selected skill、tool chain、LLM 调用状态等。这样一次回答能被复盘和审计。

### Q6：如果证据不足怎么办？

答：

> 证据不足时不会直接回答。系统会根据 gate decision 先 retry 检索；如果缺少必要槽位，比如表格问题缺指标或年份，会返回 clarify；如果重试后仍然低证据，就 refuse，避免编造答案。
